# Spec-0005: Sports Data Ingestion and Tournament Scheduler Specification

This specification details the technical migration of the Node.js scripts (`news_sync.js`, `scheduler.js`) to Kotlin Spring Boot 4.1.0 backend services, incorporating support for polymorphic sports/leagues, external cron scheduling, dynamic bracket resolutions, and user predictions scoring.

---

## 1. Internal Sync REST Controller

To support serverless execution (scale-to-zero) and prevent concurrency issues, we expose REST endpoints called by an external scheduler.

- **Endpoints**:
  - `POST /api/v1/internal/news/sync?sportId={sportId}`
  - `POST /api/v1/internal/scheduler/process?sportId={sportId}&leagueId={leagueId}`
- **Security**: Secured with OIDC signature checking. Spring Security will intercept this namespace and validate that the bearer token matches a Google Service Account linked to Cloud Scheduler.

```kotlin
package com.ligadospalpites.feature.sportsfeed.infrastructure.web

import com.ligadospalpites.feature.sportsfeed.application.usecases.SyncOrchestrator
import org.springframework.http.ResponseEntity
import org.springframework.web.bind.annotation.*
import java.util.UUID

@RestController
@RequestMapping("/api/v1/internal")
class InternalSyncController(private val syncOrchestrator: SyncOrchestrator) {

    @PostMapping("/news/sync")
    fun syncNews(@RequestParam sportId: UUID): ResponseEntity<Map<String, String>> {
        syncOrchestrator.syncNews(sportId)
        return ResponseEntity.ok(mapOf("status" to "SUCCESS", "message" to "News synced for sport $sportId"))
    }

    @PostMapping("/scheduler/process")
    fun processScheduler(
        @RequestParam sportId: UUID,
        @RequestParam leagueId: UUID
    ): ResponseEntity<Map<String, String>> {
        syncOrchestrator.syncMatches(sportId, leagueId)
        return ResponseEntity.ok(mapOf("status" to "SUCCESS", "message" to "Matches synced for league $leagueId"))
    }
}
```

## 2. Ingestion Orchestrator & Strategy Pattern

The orchestrator dynamically routes requests to the service that supports the requested sport/league.

```kotlin
package com.ligadospalpites.feature.sportsfeed.application.usecases

import java.util.UUID

interface LeagueSyncService {
    fun supports(sportId: UUID, leagueId: UUID): Boolean
    fun syncMatches(sportId: UUID, leagueId: UUID)
    fun syncNews(sportId: UUID)
}

@Service
class SyncOrchestrator(private val syncServices: List<LeagueSyncService>) {

    fun syncMatches(sportId: UUID, leagueId: UUID) {
        val service = syncServices.find { it.supports(sportId, leagueId) }
            ?: throw IllegalArgumentException("No sync service found for sport $sportId and league $leagueId")
        service.syncMatches(sportId, leagueId)
    }

    fun syncNews(sportId: UUID) {
        // Find any service supporting this sport to fetch news
        val service = syncServices.find { it.supports(sportId, UUID.randomUUID()) }
            ?: throw IllegalArgumentException("No sync service found for sport $sportId")
        service.syncNews(sportId)
    }
}
```

## 3. World Cup 2026 Ingestion Specification (`FootballWorldCupSyncService`)

This service acts as the concrete implementation for the World Cup 2026.

### API Integration Mappings:
- **Provider**: API-Sports (or fallback football-data.org).
- **Endpoint**: `https://v3.football.api-sports.io/fixtures?league=1&season=2026`.
- **Stage Mappings**:
  - `Group Stage` -> `group`
  - `Round of 16` -> `roundOf16`
  - `Quarter-finals` -> `quarterFinals`
  - `Semi-finals` -> `semiFinals`
  - `3rd Place` -> `thirdPlace`
  - `Final` -> `finalMatch`

### Localized Team Name Normalization Map:
To map English team names from the external APIs to Portuguese team names in the local PostgreSQL database, maintain a metadata lookup table (`tbl_team_name_mappings`) or a localized value map. Canonical names are generated by removing accents, spaces, and converting to lowercase:

```kotlin
fun canonicalName(name: String?): String {
    if (name.isNullOrBlank()) return ""
    return name.lowercase()
        .replace(Regex("[áàâãä]"), "a")
        .replace(Regex("[éèêë]"), "e")
        .replace(Regex("[íìîï]"), "i")
        .replace(Regex("[óòôõö]"), "o")
        .replace(Regex("[úùûü]"), "u")
        .replace(Regex("[ç]"), "c")
        .replace(Regex("[^a-z0-9]"), "")
        .trim()
}
```

## 4. Predictions Scoring Algorithm

Points are calculated when a match changes status from `inProgress` to `finished`.

### Scoring Rules:
- **Exact Score**: 25 points.
- **Correct Winner + One Correct Score**: 15 points.
- **Correct Winner/Draw Only**: 10 points.
- **Isolated Goals**: 5 points for matching a team's exact goal count.
- **Bonus multiplier**: Points are doubled ($2x$) if the match is the final (`finalMatch` stage).
- **Champion Bonus**: If the final match finishes, check special predictions. If the user successfully predicted the winner of the final stage, award **+100 points**.

### Kotlin Algorithm Implementation:

```kotlin
fun calculateMatchPoints(
    predHome: Int, predAway: Int,
    realHome: Int, realAway: Int,
    isFinal: Boolean
): Int {
    // Rule 1: Exact Score Match (25 points)
    if (predHome == realHome && predAway == realAway) {
        return if (isFinal) 50 else 25
    }

    val predWinner = when {
        predHome > predAway -> 1
        predHome < predAway -> -1
        else -> 0
    }
    val realWinner = when {
        realHome > realAway -> 1
        realHome < realAway -> -1
        else -> 0
    }

    if (predWinner == realWinner) {
        // Rule 2: Correct Winner + One Correct Score (15 points)
        if (predHome == realHome || predAway == realAway) {
            return if (isFinal) 30 else 15
        }
        // Rule 3: Correct Winner or Draw Only (10 points)
        return if (isFinal) 20 else 10
    }

    // Rule 4: Isolated Goals (5 points each)
    var points = 0
    if (predHome == realHome) points += 5
    if (predAway == realAway) points += 5
    return if (isFinal) points * 2 else points
}
```

## 5. Standings & Bracket Advancement Engine

For knockout tournaments, the sync service must resolve team placeholders dynamically in PostgreSQL:

### A. Group Stage Standings Aggregation
Sort order for teams in `tbl_group_members`:
1. Total points DESC.
2. Goal difference DESC.
3. Goals for DESC.
4. Alphabetical order ASC.

### B. Knockout Placeholders Resolution
- **Group Stage to Round of 16**: When a group (e.g. Group A) finishes (all 4 teams played 3 games), query the top 2 teams and update matches where `team_a` or `team_b` equals `1º Grupo A` or `2º Grupo A` with their real names.
- **Best 3rd Place Teams**: In a 12-group tournament, sort the twelve 3rd place teams by the standings criteria. The top 8 teams advance. Replace placeholders like `3º Grupos A/B/C/D/F` with these real team names matching their allowed group definitions.
- **Round of 16 onwards**: When a knockout match completes, locate the next match containing `Vencedor Jogo {MatchNumber}` as a team placeholder and replace it with the winner's name.
- **Match Finished Event**: When scores are finalized, publish the domain event to trigger asynchronous leaderboard caching updates:
  ```kotlin
  eventPublisher.publishEvent(
      PredictionsProcessedEvent(
          leagueId = leagueId,
          scores = scoresList // userId and points gained
      )
  )
  ```
